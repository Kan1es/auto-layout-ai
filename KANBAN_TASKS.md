# Канбан-задачи для MVP авторазметки с DART и импортом в CVAT

Документ подготовлен по `architecture.md`. Цель доски: довести проект до воспроизводимого MVP-пути:

`ZIP с изображениями -> статистика -> выбор репрезентативных кадров -> настройка DART -> авторазметка -> экспорт/импорт в CVAT`.

## Роли для команды из 5 человек

Роли можно переименовать под конкретных людей:

| Роль | Зона ответственности |
|---|---|
| Lead / Integrator | архитектурные решения, сквозная сборка, финальная демо-проверка |
| Backend | FastAPI, хранение состояния, Dataset Manager, REST API |
| ML / DART | запуск DART, нормализация результата, preview, batch inference |
| Frontend | локальный веб-интерфейс и сценарий пользователя |
| CVAT / QA / Docs | CVAT export/import, тесты, инструкции, демонстрационный сценарий |

## Рекомендуемые колонки доски

`Backlog` -> `Ready` -> `In progress` -> `Review` -> `Testing` -> `Done`

## Правила MVP

- Не делать промышленную ML-платформу.
- Не делать обучение собственной модели.
- Не делать сложную авторизацию и роли.
- Не делать Kubernetes, распределенные очереди и production-инфраструктуру.
- Не делать параллельный batch inference, если последовательная обработка уже закрывает демо.
- Не включать Qwen в обязательную готовность, пока основной путь DART -> веб -> CVAT не работает.
- В CVAT должны попадать исходные изображения и настоящая разметка, а не preview-картинки с нарисованными рамками.

## MVP-задачи

### TASK-001. Зафиксировать открытые решения MVP

**Приоритет:** P0  
**Роль:** Lead / Integrator  
**Размер:** S  
**Зависимости:** нет

**Описание:**  
Перед началом разработки нужно письменно зафиксировать решения, которые влияют на реализацию: какой именно DART используется, какой формат результата он возвращает, какие режимы поддерживаются, какой формат CVAT-импорта выбираем для bbox и segmentation, нужен ли автоматический импорт в CVAT или достаточно архива для ручного импорта.

**Что сделать:**

- Создать короткий документ `DECISIONS.md` или раздел в README.
- Ответить минимум на вопросы:
  - какой DART-репозиторий/модель используется;
  - где лежат веса и как они подключаются;
  - поддерживаются bbox, mask или оба режима;
  - prompt задает один класс или несколько;
  - bbox экспортируем в YOLO или другой формат;
  - mask экспортируем в COCO RLE/polygon или откладываем;
  - CVAT импорт делаем вручную через архив или автоматически через SDK/API;
  - максимальный размер ZIP и максимальное число изображений для MVP.

**Критерии приемки:**

- Команда понимает, какой путь является обязательным для сдачи.
- В документе нет незакрытых вопросов, блокирующих старт backend, ML и CVAT-задач.
- Если часть решений временная, это явно помечено.

### TASK-005. Реализовать безопасную загрузку и распаковку ZIP-датасета

**Приоритет:** P0  
**Роль:** Backend  
**Размер:** L  
**Зависимости:** TASK-003, TASK-004

**Описание:**  
Пользователь загружает `.zip` с изображениями. Backend должен сохранить архив, безопасно распаковать его, найти поддерживаемые изображения и подготовить рабочую папку датасета.

**Что сделать:**

- Принимать только `.zip`.
- Проверять максимальный размер архива и максимальное число изображений.
- Защититься от Zip Slip: файлы из архива не должны распаковываться за пределы папки датасета.
- Поддержать форматы `.jpg`, `.jpeg`, `.png`, `.bmp`, `.webp`.
- Игнорировать служебные файлы и неподдерживаемые расширения с warning.
- Скопировать/распаковать изображения в `images/`.
- Присвоить каждому изображению стабильный `image_id`.
- Записать `metadata.json`.

**Критерии приемки:**

- Валидный ZIP с изображениями успешно загружается.
- Поврежденный ZIP возвращает понятную ошибку.
- ZIP без изображений возвращает понятную ошибку.
- Слишком большой ZIP не валит backend.
- В metadata есть список изображений, warnings и статус датасета.

### TASK-006. Посчитать и вернуть статистику датасета

**Приоритет:** P0  
**Роль:** Backend  
**Размер:** M  
**Зависимости:** TASK-005

**Описание:**  
После загрузки пользователь должен увидеть базовую статистику по датасету: сколько изображений найдено, какие расширения, какие размеры, есть ли предупреждения.

**Что сделать:**

- Для каждого изображения прочитать width/height.
- Посчитать:
  - общее число изображений;
  - распределение по расширениям;
  - минимальный и максимальный размер;
  - часто встречающиеся разрешения;
  - warnings по битым/нечитаемым изображениям.
- Вернуть статистику через `GET /api/datasets/{dataset_id}/stats`.

**Критерии приемки:**

- Статистика корректно считается для смешанного набора JPG/PNG/WebP.
- Нечитаемое изображение не ломает весь датасет, а попадает в warnings.
- UI получает все данные одним запросом.

### TASK-008. Реализовать backend-логику отбора репрезентативных изображений

**Приоритет:** P0  
**Роль:** Backend  
**Размер:** M  
**Зависимости:** TASK-005

**Описание:**  
Пользователь задает N и вручную approve-ит изображения. Backend должен хранить историю просмотренных и список approved, чтобы пользователь мог ходить вперед/назад.

**Что сделать:**

- Реализовать endpoint-ы:
  - `POST /api/datasets/{dataset_id}/representative/init`
  - `GET /api/datasets/{dataset_id}/representative/current`
  - `POST /api/datasets/{dataset_id}/representative/next`
  - `POST /api/datasets/{dataset_id}/representative/prev`
  - `POST /api/datasets/{dataset_id}/representative/approve`
  - `POST /api/datasets/{dataset_id}/representative/unapprove`
- При `next` выбирать случайное еще не просмотренное изображение.
- При `prev` возвращаться по истории.
- Повторный approve не должен увеличивать счетчик дважды.
- Если изображений меньше или равно N, считать все изображения репрезентативными.
- Сохранять состояние в `representative.json`.

**Критерии приемки:**

- Можно выбрать N изображений через approve.
- Счетчик approved корректный.
- Назад/вперед работают предсказуемо.
- После перезапуска backend выбор не теряется.

### TASK-013. Реализовать нормализацию результата DART во внутренний формат

**Приоритет:** P0  
**Роль:** ML / DART  
**Размер:** L  
**Зависимости:** TASK-011, TASK-012

**Описание:**  
DART может вернуть нестабильный или неудобный формат. Нужно привести результат к единому internal format, который потом используют preview, autolabel и CVAT export.

**Что сделать:**

- Описать внутренний формат:
  - `image.id`
  - `image.filename`
  - `image.width`
  - `image.height`
  - `objects[].label`
  - `objects[].confidence`
  - `objects[].bbox`
  - `objects[].mask`
- Для bbox использовать абсолютные пиксели: `x`, `y`, `width`, `height`.
- Проверять, что координаты не выходят за границы изображения.
- Отфильтровывать объекты ниже `confidence`, если DART сам этого не делает.
- Сохранять raw result отдельно от normalized result.

**Критерии приемки:**

- Один и тот же internal format используется для bbox и mask.
- Невалидные координаты не попадают молча в результат.
- Внутренний формат не зависит напрямую от CVAT.
- Есть примеры normalized JSON.

### TASK-015. Реализовать API настроек DART и preview-запуска

**Размер:** L

**Критерии приемки:**

- Пользователь может поменять prompt/confidence/mode и получить новый preview.
- Последние настройки сохраняются.
